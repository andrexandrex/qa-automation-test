Conclusiones API - PetStore
===========================

La prueba API valida correctamente el flujo solicitado para mascotas en
PetStore. El escenario no se limita a revisar codigos de estado: tambien
verifica que la informacion creada y modificada sea la que devuelve el
servicio.

Un punto importante fue separar la URL principal de PetStore del prefijo de la
API. La pagina https://petstore.swagger.io funciona como entrada y
documentacion, pero los endpoints REST se consumen bajo /v2. Si se intenta
llamar directamente a /pet sin ese prefijo, el servicio responde 404 porque esa
ruta no existe en la raiz del sitio.

Durante la ejecucion en GitHub Actions tambien se observo que el endpoint
publico /v2 puede rechazar peticiones POST desde el runner con codigo 403. Para
evitar que la validacion dependa de una restriccion de infraestructura externa,
el workflow permite sobrescribir la URL de API y usa el endpoint compatible
https://petstore3.swagger.io/api/v3. El escenario funcional se mantiene igual.

Despues de revisar un reporte mas reciente, se confirmo que el bloqueo no
dependia solamente de la version del endpoint. GitHub Actions tambien recibia
403 al usar el endpoint v3. La causa mas probable era que el servicio rechazaba
peticiones con la firma por defecto de python-requests desde los runners de
GitHub. Por eso se agregaron headers explicitos de cliente HTTP, incluyendo
Accept, Content-Type y un User-Agent similar al de un navegador. Con ese ajuste
la prueba API paso correctamente en la rama de correccion creada para validar
el problema antes de mezclarlo a main.

La prueba usa datos unicos en cada ejecucion. Esto es necesario porque PetStore
es una API publica: otras personas o ejecuciones anteriores pueden dejar datos
en el servicio. Usar IDs y nombres generados reduce la posibilidad de falsos
fallos por colisiones.

Tambien se agregaron reintentos en las verificaciones. La razon es que el
servicio puede mostrar pequenas demoras entre una accion y la disponibilidad
del dato al consultarlo. El enfoque usado mantiene las acciones principales
claras y evita repetir operaciones como crear o actualizar; solo se repite la
lectura hasta confirmar el resultado.

El reporte HTML por timestamp ayuda a revisar historicos sin perder evidencia.
Cada corrida API deja un archivo nuevo en reports/api/, lo que permite comparar
resultados entre ejecuciones locales o descargas de GitHub Actions.

La conclusion principal del lado API es que, al trabajar contra servicios
publicos de demostracion, no basta con validar que el endpoint funcione desde
la maquina local. Tambien hay que observar como responde desde CI, porque los
runners compartidos pueden ser tratados de forma distinta por el servidor. El
proyecto quedo preparado para eso mediante configuracion por variable de
entorno, mensajes de error mas descriptivos y reportes descargables desde
GitHub Actions.
